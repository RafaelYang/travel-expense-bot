## Project learnings

- The app is a Next.js 16 + Prisma/PostgreSQL shared travel-expense tracker with Google/LINE authentication, trip invitations, exchange-rate conversion, image attachments, and a LINE Messaging API bot.
- Verified checks from the project root: `npm test` passes 42 security/money/image/reconciliation/statistics/exchange-rate/PWA/realtime-version regression tests, `npx tsc --noEmit` passes, `npm run lint` succeeds with no errors (9 existing image performance warnings), and `npm run build` succeeds on Next.js 16.2.6. There is no `prisma/migrations` history yet.
- Treat expense and deposit authorization as a child-resource invariant: every read/update/delete must bind the child ID to its `tripId`, then verify the acting user is an eligible trip member. LINE postbacks must additionally bind the sending `lineUserId` to that same trip before mutating data.
- Reuse `src/lib/trip-access.ts` for expense/deposit mutations. Writable roles are `owner` and `member`; `viewer` is read-only. LINE expense mutations additionally require the sender to own the expense.
- Expense image delivery is intentionally sessionless for LINE servers but requires the expiring HMAC query parameters produced by `src/lib/expense-image-signing.ts`. Web DTOs replace stored image values with 24-hour signed references, so Base64 data no longer enters initial RSC/trip API payloads; PATCH resolves valid references back to stored values. Production should set `IMAGE_URL_SIGNING_SECRET`; the current fallback order is `AUTH_SECRET`, then `LINE_CHANNEL_SECRET`. Keep the response private/no-store and never restore unsigned ID-only URLs.
- `LineBotState.activeTripId` currently stores either `tripId` or `tripId:currency`; consumers must split the value before using it as a Prisma trip ID. Prefer separate typed columns when the schema is next migrated.
- Monetary values currently use `Float`. `src/lib/money.ts` deliberately excludes missing foreign conversions and foreign-currency deposits from base-currency totals; never silently substitute a 1:1 rate. A future schema migration should use exact decimal/minor-unit values and persist deposit conversions consistently.
- Web expense and deposit amount controls must use `type="number"`, `inputMode="decimal"`, and `step="any"`; the HTML default step is 1 and will block decimal submission with a native `stepMismatch` before the positive-number API schemas run. Keep this contract aligned across create and edit fields; `tests/money-input.test.ts` guards the three current form locations.
- Expense reconciliation preserves the booking-time `convertedAmount` and `exchangeRate`, stores a foreign-card final charge separately in nullable `settledAmount`, and uses it in totals only while `reconciledAt` is set. Foreign-card confirmation requires a positive final charge; changing amount, currency, or payment method clears both reconciliation fields unless the same PATCH explicitly re-confirms with a valid new final charge. LINE money and currency edits must clear both fields too.
- Homepage batch reconciliation intentionally lists only unreconciled card expenses and reuses the authorized per-expense PATCH route. Foreign-card rows require a positive actual base-currency charge; base-currency card rows send only `reconciled: true`. Keep partial success retryable with `Promise.allSettled`, one full-trip refresh after the batch, and nullable API money fields handled without formatting `null`.
- Expense images and custom cover images are still stored as Base64 in database fields. Expense images are now omitted from initial client payloads, but future storage work should move the underlying blobs and custom covers to authenticated object storage/thumbnails. LINE image ingestion lives in `src/lib/line-expense-image-service.ts`, enforces a 5 MB/type limit, and uses the previous JSON value as an optimistic concurrency guard.
- Exact-money migration is intentionally staged, not applied: `prisma/manual-migrations/20260714_money_decimal_expand.sql` adds/backfills shadow Decimal columns and has a paired rollback script. Because the live schema has no Prisma migration history, never move these files under `prisma/migrations` or run them automatically; follow the adjacent README and validate on staging before dual-write code is introduced.
- This project uses the Next.js 16 `proxy.ts` convention for optimistic cookie redirects. Keep authoritative authentication and authorization in route handlers/data-access code rather than relying on the proxy.
- Keep the homepage data path server-first: `src/app/page.tsx` authenticates and calls `src/lib/trip-dashboard.ts`, while `src/app/home-client.tsx` only owns interactions. Expense/deposit POST results are merged directly into `TripDetailClient` totals and lists instead of triggering a full-trip GET; the Navbar deduplicates LINE status reads with a 60-second client cache.
- Cross-device trip updates use `GET /api/trips/[tripId]/version`: while the page is visible, the client checks the member-authorized transaction fingerprint every 5 seconds and fetches the full trip only when it changes, with an immediate check on focus/visibility return. Keep `src/lib/trip-version.ts` ordering-independent and include expense `updatedAt` plus deposit value fields because deposits have no `updatedAt` column.
- Foreign cash is a per-trip, per-user, per-currency wallet. A `buy` exchange adds foreign cash and counts its base-currency amount as spending; cash-paid expenses only debit the wallet and are excluded from net-spending totals; a `sell` exchange debits foreign cash and reduces net spending by the base amount received. Keep wallet changes and expense/exchange writes in one Prisma transaction, including LINE edit/delete paths, and include cash wallets/exchanges in the realtime fingerprint.
- The trip timeline is a discriminated union of expenses, deposits, and cash exchanges. Both the Trip RSC and `GET /api/trips/[tripId]` must serialize `summarizeTripSpending().exchangeNet` so the UI can state how exchanges affect the total. Exchange rows use their own editor and are editable only by their creator; amount edits apply the signed wallet delta and an all-old-values optimistic guard in one transaction. Include exchange currency, rate, date, and note in the realtime fingerprint because `CashExchange` has no `updatedAt`.
- Trip statistics are centralized in `src/lib/trip-statistics.ts`: daily fund flow is non-cash spending plus buys minus sells, while category consumption includes card and cash expenses but excludes exchanges. Keep pre-trip/trip/all scopes aligned across both views, preserve missing-conversion counts, fill only elapsed trip days (never future padding), and retain the category-to-expense-to-detail drill-down with scroll/focus restoration. Regression coverage is in `tests/trip-statistics.test.ts`. Trip boundary dates use their stored calendar-day prefix; transaction day grouping remains browser-local until the schema persists an explicit trip time zone or transaction calendar day.
- Reference exchange rates use CurrencyBeacon first with a 30-minute latest-quote cache, finite-positive validation, and a 6-second provider timeout; the daily open ExchangeRate-API fallback is cached for 24 hours. Seven quoted-day trends come only from same-source CurrencyBeacon timeseries data cached for 6 hours, never from mixed or fabricated history. The authenticated proxy is limited to `ALL_CURRENCIES`, 30 requests per user per minute, and a 128-entry history cache. Keep the UI explicit that these are reference estimates; actual cash exchanges remain user-entered and card totals remain reconciliation-driven. Headline rounding is presentation-only: retain the raw quote for calculations and history, and keep each displayed amount plus currency code together as a no-wrap unit across desktop and mobile widths. Keep the calculator result concise by showing the estimate without repeating the arithmetic derivation underneath.
- The global Traditional Chinese UI font is self-hosted `Huninn` loaded through `next/font`, with 17 px mobile and 18 px desktop root sizing. Keep native controls inheriting the font, and use `--color-primary-text` or primary text colors for labels on dark surfaces instead of raw category/background colors.
- Manual trip edit/share/exchange/deposit overlays use `ModalScrollLock` backed by `react-remove-scroll`; keep `trip-modal-scroll-area` on inner overflow containers so the modal scrolls without moving the page and closing restores page scrolling. Category, payment, and currency selectors share the opaque `trip-choice-button` states with `aria-pressed`, group labels, and visible focus rings; do not restore transparent unselected controls.
- The cash-wallet expand schema in `prisma/manual-migrations/20260716_cash_wallet.sql` was applied to production on 2026-07-16 after `node scripts/cash-wallet-migration.mjs` completed a transactional dry run and verified rollback; `--apply` then passed the post-migration data checks. The paired rollback is destructive once wallet data exists. At that release, Vercel Preview had no `DATABASE_URL`, so verify preview database isolation again before relying on it for future migrations.
- The app is installable as a standalone PWA through `src/app/manifest.ts`, public 192/512/maskable/Apple icons, and the global `InstallAppPrompt`: Chromium uses the native install prompt, while Apple devices receive platform-specific instructions. Keep `/manifest.webmanifest` and `/images/pwa/*` outside proxy authentication redirects. Do not add generic service-worker caching for authenticated trip/API/RSC data; introduce offline support only with an explicit user-scoped cache and invalidation design.
- Production is the Git-integrated Vercel project `travel-expense-bot`; pushes to GitHub `RafaelYang/travel-expense-bot` branch `main` deploy automatically to `https://travel-expense-bot-steel.vercel.app`. Verified post-deploy checks are `/login` = 200, unauthenticated `/` = 307 to `/login`, unauthenticated trip-version API = 401, plus a recent production error-log scan. The staged manual Decimal SQL files are documentation/runbooks only and must not be executed during ordinary app deployment.
- The user explicitly authorizes this project to be deployed after requested changes pass the relevant checks, so complete the Git-integrated production deployment and post-deploy verification instead of stopping at an uncommitted local result. Database-changing releases must still follow the migration safeguards above.
